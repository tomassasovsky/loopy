# ASIO buffer/sample-rate count clamp before array indexing

## What We're Building

`NativeAudioEngine.enumerateAsioDrivers` (packages/loopy_engine/lib/src/native_audio_engine.dart, ~line 226) copies ASIO buffer sizes and sample rates out of a native `le_device_info` struct into Dart lists. The native struct has fixed 8-element C arrays (`asio_buffer_sizes[8]`, `asio_sample_rates[8]`, declared in packages/loopy_engine/src/core/loopy_engine_api.h and mirrored in the generated FFI bindings as `@ffi.Array.multi([8])`), but the accompanying counts (`asio_buffer_count`, `asio_sample_rate_count`) are used directly as loop bounds with no clamp. `dart:ffi`'s `Array<T>` does no bounds checking, so a native build that ever reports a count > 8 would silently read adjacent struct memory as bogus buffer sizes / sample rates instead of crashing or erroring.

Today this is unreachable — `win_asio_device.cpp` loops with an explicit `n < 8` guard on both fill functions, and the non-Windows stub always reports 0 — but the Dart side has no defense of its own, which is inconsistent with this same file's existing defensive clamp for the analogous `pluginStateGet` case (~line 530-536), which explicitly guards against a native count exceeding an allocated buffer.

The fix: clamp both counts to the array size before using them as loop bounds, matching the established style/comment convention already in this file.

## Why This Approach

**Chosen approach: local named constant + ternary-min clamp, matching the existing `pluginStateGet` pattern.**

Considered alternatives:

1. **Introduce `_maxAsioSlots = 8` constant in native_audio_engine.dart, clamp both counts with the same `x < y ? x : y` ternary-min style already used at line 535, with an explanatory comment.** (Chosen)
   - Pros: Matches existing in-file convention exactly (reviewers/future maintainers see one consistent defensive idiom in this file, not two different clamp styles). No dependency on generated code. Self-documenting via comment tying the constant back to the header's literal `8`.
   - Cons: The `8` is still duplicated (header literal + generated binding annotation + this new Dart constant) — but that duplication already exists between the header and the generated bindings today, so this doesn't make things structurally worse, and there is no existing named constant anywhere in the codebase to reuse.
   - Best when: keeping the fix narrowly scoped to one file, consistent with sibling code, and not touching generated/native code.

2. **Use `.clamp(0, 8)` inline at each loop-bound call site, no named constant.**
   - Pros: Slightly less code.
   - Cons: Magic number `8` with no explanation of where it comes from; doesn't match the existing named/commented style in this file; less obvious that it's tied to the array size specifically (as opposed to some arbitrary limit).

3. **Add a shared constant to the generated bindings or the C header and reference it from Dart.**
   - Cons: Generated bindings are regenerated by ffigen (per `loopy-ffigen-format-drift` memory) and hand-editing them is fragile/would be overwritten; touching the C header is out of scope for a Dart-side defensive fix and drags in a rebuild of native code, which is disproportionate to a low-severity, already-unreachable defensive issue. Rejected as over-scoped for this fix.

Approach 1 wins: minimal, in-file, consistent with precedent, and does not touch generated or native code.

## Key Decisions

- **Clamp style**: reuse the exact `final n = written.value < size ? written.value : size;`-style ternary-min already established in this file (not `.clamp()`), for stylistic consistency within the same file. Applied as `final bufferCount = ... < _maxAsioSlots ? ... : _maxAsioSlots;` (and the sample-rate equivalent), then used as the loop bound.
- **Constant name/placement**: a single private constant `_maxAsioSlots = 8` defined near the top of `native_audio_engine.dart` (or immediately above `enumerateAsioDrivers`), with a comment noting it mirrors `asio_buffer_sizes[8]` / `asio_sample_rates[8]` in `loopy_engine_api.h` — reused for both the buffer-size and sample-rate clamps since both arrays are the same fixed size.
- **No lower-bound clamp needed beyond what's natural**: the existing loop already starts at 0 and uses `<`, so a native count that's simply not negative and not over 8 works correctly; if a negative count somehow arrived, the `for` loop naturally produces an empty list (condition `0 < negative` is false), so only the upper bound needs an explicit clamp. (Documented assumption: negative counts are already implicitly safe under Dart's `for` loop semantics, so we only need to protect against the upper bound.)
- **No test added**: confirmed no existing seam constructs a fake native `le_device_info` struct with an out-of-range count (no `native_audio_engine`/`asio` test files exist in packages/loopy_engine/test/). Per the issue's own guidance, shipping the defensive clamp without a dedicated test is acceptable here. This is a documented, deliberate trade-off, not an oversight.
- **Scope discipline**: touch only the two loop-bound expressions (and the new constant) in `enumerateAsioDrivers`; do not refactor `_enumerate`, other methods, or generated/native code, per the parallelized-fix constraint (1 of 21 independent worktree fixes).
