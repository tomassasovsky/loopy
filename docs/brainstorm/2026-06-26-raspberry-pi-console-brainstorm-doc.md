# Raspberry Pi Loopy — All-in-One Floor Console

## What We're Building

A self-contained, standalone Loopy looper built around a **Raspberry Pi 5**, packaged as a single rugged **floor console**: a tilted enclosure with two built-in screens up top (a **16″ main UI** and a **7″ waveform** display) and the foot controls — footswitches, rotary encoder, and LEDs — on the front edge where they can be stomped. A **USB class-compliant audio interface** provides instrument/mic input and stereo output. The unit boots straight into the existing Loopy Flutter app in kiosk mode; no laptop, no separate pedal, no USB-MIDI bridge.

The headline finding from the codebase review: **this is mostly an integration job, not a rewrite.** Loopy's Linux build is already a first-class, CI-green target running the exact PipeWire→JACK→ALSA audio path Pi OS uses; the waveform already renders to a second OS window purpose-built for a second physical display; and the control layer already exposes a `ControllerSourceKind.gpio` seam *literally labeled "A Raspberry Pi GPIO pin edge"* ([`controller_input.dart:12`](packages/controller_repository/lib/src/controller_input.dart:12)). The only genuinely new software is one `gpio_client` package and a kiosk launcher. The bulk of the effort is hardware/enclosure and a small dedicated LED driver.

## Why This Approach

Three hardware forks were decided collaboratively; each chose the option that maximizes reuse of proven Loopy code and minimizes new real-time/timing risk:

- **Form factor → all-in-one floor console** (over desktop command-center or modular brain-box). A 16″ screen implies a tabletop/standing unit; integrating everything into one tilted enclosure keeps all control wiring internal and short (ideal for GPIO) and makes it a true performance instrument.
- **Controls → hybrid GPIO + dedicated LED driver** (over all-direct-GPIO or reusing the 32U4 board). Footswitches and the encoder are trivial, reliable digital GPIO inputs. The WS2812 LEDs are the *one* finicky part — they need microsecond-precise timing the Pi handles awkwardly (SPI/PWM+DMA, root config, 3.3→5 V level shift, Pi-5 driver quirks). Offloading just the LEDs to a tiny driver keeps that timing off both the Pi and the audio path, which is the right reliability trade for a live unit, while still honoring "controls through GPIO" for everything else.
- **Audio → USB class-compliant interface** (over an I2S HAT). It reuses Loopy's Linux audio path with **zero new code** — the engine already enumerates, selects, and auto-JACK-pins a chosen USB interface ([`engine_devices.c`](packages/loopy_engine/src/core/engine_devices.c), [`engine_linux.c:156`](packages/loopy_engine/src/platform/engine_linux.c:156)), and the repo even ships Focusrite setup notes ([`docs/RUNNING_ON_LINUX.md`](docs/RUNNING_ON_LINUX.md)). It also leaves **every GPIO pin free** for the controls (an I2S HAT would consume the I2S pins) and gives proper instrument/mic I/O. USB interfaces hit the ≤10 ms round-trip latency target fine.

## Key Decisions

- **Compute: Raspberry Pi 5.** Driving two displays + real-time low-latency audio + a Flutter/Skia UI needs the Pi 5's GPU and headroom; Pi 4 is marginal for dual-display + audio. Pi 5's RP1 I/O also gives the dual micro-HDMI used below. (Plan for active cooling — the Pi runs warm in a closed enclosure under sustained audio + GPU load.)
- **Displays: both over HDMI** (Pi 5 has 2× micro-HDMI). 16″ portable HDMI monitor = main UI ([`BigPictureView`](lib/looper/view/big_picture_view.dart:26)); 7″ HDMI display = waveform ([`WaveformView`](lib/visualizer/widgets/waveform_view.dart:10)). The kiosk compositor must map the waveform's second OS window (opened by [`WaveformWindowService`](lib/visualizer/waveform_window_service.dart:29) via `desktop_multi_window`, pushed ~30 fps) onto the 7″ output. *Open: confirm the 7″ is HDMI vs the official DSI ribbon — see Open Questions.*
- **Audio: USB class-compliant interface, no engine changes.** Reuses miniaudio + the Linux JACK-pinning path verbatim. Pick an interface with real instrument + mic inputs (Scarlett-class). PipeWire "Pro Audio" profile + the existing `PIPEWIRE_QUANTUM` forcing ([`engine_linux.c:224`](packages/loopy_engine/src/platform/engine_linux.c:224)) tunes buffer size.
- **Footswitches + encoder: direct Pi GPIO via a NEW `gpio_client` package.** This is the single new native binding (libgpiod). It implements the existing `ControllerSource` interface and emits `RawControllerInput` into the unchanged `ControllerRepository → ControllerMapping.resolve() → ControllerEvent → LooperBloc` pipeline — so **no changes to the engine, MIDI stack, BLoC layer, UI, or theme.** Footswitches = one GPIO each (pull-up + software debounce, leading-edge for low latency — see the deferred debounce note in memory); encoder = A/B/SW pins (software quadrature; the Pi has no hardware decoder, fine at human turn speed).
- **LEDs: tiny dedicated WS2812 driver, fed by the Pi.** A small helper (a cheap RP2040/QT-Py, or the existing 32U4 you already designed) receives compact "LED state" messages from the Pi (SPI or UART) and runs the WS2812 ring + indicator strip with FastLED-grade timing. This isolates the only hard-real-time piece. *Open: exact driver chip + Pi↔driver protocol — see Open Questions.*
- **Software: kiosk appliance, reuse the Flutter Linux runner.** Boot straight into the app full-screen across both displays (GTK on Wayland, or evaluate `flutter-pi`). The GTK runner already forces the Skia renderer for correct Material icons ([`linux/runner/main.cc:32`](linux/runner/main.cc:32)).
- **New work is bounded and known:** (1) the `gpio_client` package + a Pi-specific `ControllerMapping` (GPIO pin → `LooperEvent`); (2) the LED-driver firmware + Pi-side LED-output channel; (3) an **ARM64 build/CI job** (none exists today); (4) a kiosk/dual-display launcher + boot config; (5) the physical enclosure. Explicitly **out of scope / unchanged:** audio engine, MIDI stack, all BLoCs, the entire UI, and the theme system. **Heaviest two non-app chunks (don't under-weight in planning):** the **enclosure** (tilt-mounting two screens + a stompable footswitch panel + ruggedization) and **kiosk/dual-display boot integration** (Wayland compositor config, output naming, auto-launch, second-window pinning) — both are larger than a single line implies.
- **Configuration input is unresolved and gating — needs a decision.** The foot controls cover *performance* (rec/play/stop/track/bank), but the app's *setup* surfaces — audio-device selection, track rename, the FX/routing panel — live in `BigPictureSettingsPage` / `SignalListView` and today need a pointer/touch. A no-pointer console can't reach them. Options to weigh in planning: (a) make the **16″ a touchscreen** (simplest, reuses the existing touch UI as-is); (b) drive config via the **encoder as a menu system** (new UI work, but keeps the unit pointer-free); (c) **config-over-network** (phone/laptop hits the unit for setup, foot-only in performance). Affects the 16″ hardware BOM.
- **Relationship to the existing pedal: a parallel variant, not a replacement.** The ATmega32U4 USB-MIDI pedal (PR #85) stays valid for "laptop + pedal" users; the Pi console is a second, standalone product line that shares 95% of the app/engine.

## Open Questions

- **7″ display interface — HDMI vs DSI?** HDMI keeps both screens on one bus type and maps cleanly to the second-window service, but uses the Pi's second micro-HDMI. The official 7″ DSI display frees an HDMI port and is mechanically tidy, but is low-res (800×480) and needs DSI compositor mapping. Resolution matters less for a waveform — worth a quick spike.
- **LED driver chip + Pi↔driver protocol.** RP2040/QT-Py (clean, cheap, USB-flashable, PIO-perfect for WS2812) vs reusing the already-designed 32U4 (zero new firmware if we keep its existing LED path). And: SPI vs UART vs a minimal USB-serial link for pushing LED state from the Pi.
- **Kiosk rendering target: GTK-on-Wayland vs `flutter-pi`.** `flutter-pi` is lighter/faster on the Pi and boots to fullscreen, but `desktop_multi_window` and `window_manager` assume a desktop GTK environment — does the two-window waveform setup work under `flutter-pi`, or does GTK-on-Wayland (heavier but compatible) win? Needs an on-device spike.
- **Dual-display window placement.** How reliably can we pin the waveform's second OS window to the 7″ output at boot (compositor config, `wlr-randr`, output names)? Validate the ~30 fps second-window path on real dual outputs (also a standing roadmap item).
- **Per-display UI scaling / DPI.** The 16″ (likely 1080p) and 7″ (possibly 800×480) differ sharply in size and density. Confirm the main UI and waveform window each get a sensible Flutter logical-pixel scale per output under the kiosk compositor, so neither is unusably tiny or huge.
- **GPIO input protection.** Pi GPIO is **3.3 V and not 5 V-tolerant**. Footswitch/encoder lines (longer internal runs, ESD, contact bounce) need series resistors / clamping so a miswire or transient can't kill a Pi pin. Spec this in the controls wiring.
- **Boot/power resilience.** A stompable performance unit will get its power yanked mid-session. Plan **read-only rootfs (or overlayfs) + a safe-shutdown story** so a hard power-cut doesn't corrupt the SD card / boot.
- **Thermals & power.** Single power input for Pi 5 + two screens + USB interface + LED driver; budget the supply and confirm active cooling keeps the Pi from throttling during long sessions.
- **On-hardware latency gate.** Re-run the ≤10 ms round-trip target on the chosen USB interface + Pi 5 + PipeWire quantum before committing to the interface model.
