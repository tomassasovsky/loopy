## Clamp ASIO buffer/sample-rate counts before array indexing - Minimal

`NativeAudioEngine.enumerateAsioDrivers` in
`packages/loopy_engine/lib/src/native_audio_engine.dart` (~line 226) copies
native `le_device_info.asio_buffer_sizes`/`asio_sample_rates` (fixed 8-element
C arrays, `@ffi.Array.multi([8])` in the generated bindings) into Dart lists,
using the native `asio_buffer_count`/`asio_sample_rate_count` fields directly
as loop bounds with no upper-bound check. `dart:ffi`'s `Array<T>` does no
runtime bounds checking, so a native count > 8 (never expected today, since
`win_asio_device.cpp` hard-guards both fill loops at `n < 8` and the
non-Windows stub always reports 0) would silently read past the 8-slot array
into adjacent struct memory instead of erroring. This file already has an
established defensive-clamp pattern for the exact same class of problem in
`pluginStateGet` (~line 530-536). Add the same style of clamp here.

## MVP

Add a private constant near the top of the class (or immediately above
`enumerateAsioDrivers`):

```dart
// ASIO buffer sizes/sample rates are fixed 8-element arrays on the native
// side (asio_buffer_sizes[8] / asio_sample_rates[8] in loopy_engine_api.h).
// Clamp the native-reported counts to this before using them as loop bounds
// so an out-of-range count can never read past the array (mirrors the
// defensive clamp in pluginStateGet above).
static const _maxAsioSlots = 8;
```

Then in `enumerateAsioDrivers`, clamp both counts before the list literals:

```dart
final bufferCount = (outPtr + i).ref.asio_buffer_count;
final clampedBufferCount =
    bufferCount < _maxAsioSlots ? bufferCount : _maxAsioSlots;
final rateCount = (outPtr + i).ref.asio_sample_rate_count;
final clampedRateCount =
    rateCount < _maxAsioSlots ? rateCount : _maxAsioSlots;
```

And use `clampedBufferCount`/`clampedRateCount` as the `for` loop bounds
instead of the raw `(outPtr + i).ref.asio_buffer_count` /
`asio_sample_rate_count` reads, inside the `AudioDevice(...)` constructor
call's `bufferSizes`/`sampleRates` list literals.

(Exact variable naming/structure to be finalized during implementation to fit
cleanly inside the existing list-literal-in-constructor-argument style; the
above is illustrative, not prescriptive line-for-line.)
